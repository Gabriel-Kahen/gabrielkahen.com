The Scenario

You're building a dashboard for a SaaS product.

Three things need to happen:

  • Fetch user data from an API
  • Query analytics from a database
  • Generate a usage report

These could start right away. But before you code, what do you need to know?

Step 1: Gathering Context
Pattern: Understanding Context First

Before acting, you ask:

  • How quickly does this need to load? (affects architecture)
  • Are these API calls independent? (affects sequencing)
  • What if one fails? (affects error handling)
  • How often will this data change? (affects caching)

Why this matters: Jumping to code without answering these creates brittle solutions. Understanding shapes the entire approach.

Step 2: Recognizing Independence
Pattern: Parallel Operations

Key realization: None of these three tasks depend on each other's output.

  • User data doesn't require analytics first
  • Analytics doesn't require user data
  • The report can generate from both in parallel

Decision: Execute all three simultaneously instead of chaining them. This reduces load time and mirrors how independent systems naturally work.

Promise.all([ fetchUserData(), queryAnalytics(), generateReport() ])
Step 3: Investigating Edge Cases
Pattern: Investigate Before Claiming

Uncertainty emerges: What if the API times out but the database responds?

Two approaches compete:

  • A: Show a partial dashboard (user + analytics, no report)
  • B: Show nothing until all three succeed

Rather than guess, you investigate the requirements again: Is partial data useful or confusing? How often do these failures actually occur? What does the user expect?

Conclusion: Approach A is better. Partial data is useful. Timeouts are rare but possible. Show what loaded, indicate what's missing.

Step 4: Honest Assessment
Pattern: Honesty Over Validation

Building it out, you notice: Partial loading is complex to handle. Error states multiply.

You could hide this complexity—ship something that "works." Or you could name the problem:

"This is more brittle than ideal. If we need robustness, we should implement explicit loading states and retry logic, not hide failures."

This honesty shapes the decision: Do we add the extra complexity now, or document the risk for later?

Action: Add explicit loading states. It's the right answer, even though it takes longer.

Step 5: Documenting the Path
Pattern: Documenting Process

Work is done. Code is shipped. But you're not finished.

You document:

  • Why parallel execution was chosen (not sequential)
  • Why partial loading was accepted (investigation revealed it was right)
  • Where brittleness remains (loading state complexity)
  • What would improve it (retry logic, backoff strategy)

Why this matters: Future you, or someone else, can understand the reasoning, not just the code. When requirements change, the thought process guides the adaptation.
